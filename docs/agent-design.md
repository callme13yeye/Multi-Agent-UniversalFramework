# TK-MultiAgent — Agent 系统设计文档

## 一、概述

TK-MultiAgent 是一个企业级多智能体协作系统，核心能力是让大语言模型自主完成复杂任务。系统采用**两层 DeepAgent 编排架构**，将任务理解与任务执行分离，通过领域专家智能体（Specialist Sub-Agent）的分工协作、人机协同的审批机制、以及智能模型网关的健康管理，实现可靠的长周期任务自主执行。


---

## 二、核心理念

### 2.1 LLM 自路由

系统不依赖外部分类器或规则引擎来决定调用哪个工具或专家。所有可用的工具描述和专家能力列表始终在模型的系统提示词中可见，模型通过函数调用能力自行判断和选择。这种设计避免了硬编码路由规则带来的维护负担，同时让模型能够根据上下文做出灵活的判断。

### 2.2 关注点分离

每一层有明确的职责边界：

- **分流层（Triage）**：只做一件事——判断任务复杂度。简单任务直接处理，复杂任务则创建后台任务交给执行引擎。
- **执行层（Executor）**：负责复杂任务的完整生命周期——制定计划、逐步委托专家、根据结果动态调整、处理审批、汇报成果。

这种分离使得每层的系统提示词可以保持精炼，模型不容易混淆职责，同时架构上各层可以独立演化。

### 2.3 热插拔与零停机演进

系统中的一切——工具、专家智能体、模型——都支持运行时热替换，无需重启服务。新增一个专家只需创建一个目录和一份描述文件，系统自动发现并注入。

## 三、两层 DeepAgent 架构

### 3.1 整体架构

用户请求进入系统后，首先经过**限流中间件**和**网关中间件**，然后进入 **Agent 中间件管线**（包含模型调用限制、智能路由、重试、工具调用限制等），最终到达 Triage DeepAgent。

Triage 做出分流判断：
- 简单任务 → 直接调用工具或委托一次 Specialist → 返回结果
- 复杂任务 → 创建后台任务 → 立即返回任务编号 → 后台 Executor 接管执行

后台 Executor 在执行过程中通过**事件总线**实时推送状态变化，前端可以通过 SSE 流订阅任务进度。

### 3.2 Triage DeepAgent（分流层）

Triage 是用户对话的入口。它的系统提示词约定了清晰的分流规则：

**简单任务（直接处理）**需同时满足：单个专家能完成、不涉及审批、单轮可给出完整答案。

**复杂任务（创建后台任务）**满足任一即触发：需要多个专家协作、涉及审批或人类决策、数据量大需分批处理、跨系统长周期跟踪。

Triage 持有系统的全部工具（知识检索、搜索、时间、任务管理等），可以根据场景灵活选择。当它把任务委托给某个 Specialist 后，如果收到的回复中包含人审标记，说明这个任务实际上涉及审批流程——Triage 会立即将任务升级为后台任务，由 Executor 接管审批流程。

### 3.3 Executor DeepAgent（执行层）

Executor 是后台任务的执行引擎。它与 Triage 使用同一类型的 DeepAgent 实例，区别在于系统提示词和工具配置不同：

- **系统提示词**引导它按"分析任务 → 逐步执行 → 处理审批 → 汇报结果"的流程工作
- **工具集**更精简：只有委托专家（task）、发起审批（request_approval）、读取执行日志（read_task_journal）这三个工具

Executor 的核心行为模式是"逐步执行，边做边看"——它不会一次性规划所有步骤然后盲目执行，而是每完成一步、看到结果后，再决定下一步做什么。这种增量式执行策略让模型能够根据中间结果动态调整计划，而不是死守初始方案。

## 四、Specialist Sub-Agent（领域专家）

### 4.1 设计理念

每个领域专家是一个独立的智能体，拥有自己的角色身份、领域知识和专属工具。它们不是被硬编码的，而是通过文件系统自动发现和加载的。

### 4.2 定义方式

每个 Specialist 对应 `app/subagents/` 下的一个子目录，目录中只需要一个 `AGENT.md` 文件。这个文件使用 YAML 前置元数据声明专家的名称、能力描述、允许使用的工具列表，以及可选的输出结构约束；正文则是该专家的系统提示词。

前置元数据的关键字段：
- **name**：专家名称，供路由层引用
- **description**：一句话能力描述，直接出现在模型的工具选择列表中——模型据此判断该委托给谁
- **allowed_tools**：该专家可以使用的工具名称列表，从全局工具注册中心按名索取
- **output_schema**（可选）：约束专家的输出格式，支持内联 JSON Schema 或引用已注册的 Schema 名称

### 4.3 自动发现与注入

系统启动时（以及热加载时），自动扫描 `app/subagents/` 目录，解析每个子目录下的 `AGENT.md`，构建 Specialist 列表。这个列表通过 SubAgent 中间件注入到 Triage 和 Executor 的系统提示词和可用工具中，使模型可以像调用工具一样委托专家。

### 4.4 工具隔离

每个 Specialist 只获取其声明需要的工具，而非全部工具。这样可以显著减少模型上下文中的工具描述占用的 token 数量，让模型更专注于自己的领域。

## 五、工具系统

### 5.1 注册中心

所有工具通过一个全局注册中心管理。工具定义时使用装饰器标注，自动进入注册中心。工具按来源模块自动分类（通用工具、知识检索、人审确认、任务管理等），生成系统提示词时按分类组织，让模型更容易理解工具的适用场景。

### 5.2 工具分层

不同层级的 Agent 持有不同的工具集：

- **Triage** 持有全部工具——它需要处理各种简单任务，工具越多越灵活
- **Executor** 只持有三个工具——它的职责是编排而非执行，工具少反而让它更专注
- **Specialist** 各持所需——只拿自己领域相关的工具

### 5.3 工具热加载

`app/tools/` 目录下的文件变更会被自动检测。当工具文件新增或修改时，热加载器触发 Agent 重建流程：重新收集工具、重新生成提示词、重新创建 Agent 实例，最后原子性地替换运行中的实例。

## 六、Agent 中间件管线

DeepAgent 在创建时自动组装一条中间件管线。每一层中间件负责一个横切关注点，模型调用按顺序穿过这些中间件。基础管线由 DeepAgents 框架提供，项目在此基础上注入领域定制的中间件。

管线结构如下（按执行顺序）：

### 6.1 模型调用限制（ModelCallLimitMiddleware）

防止单个请求或整个会话消耗过多模型调用。分别设置单次运行上限和整个对话线程上限。达到上限时的行为可配置（当前为优雅结束）。

### 6.2 智能网关路由（GatewayMiddleware）

这是项目定制的核心中间件。它不采用简单的"主模型失败→切换备用模型"的静态降级，而是通过智能网关实现健康感知路由：
- 实时跟踪每个模型的健康状态（延迟、成功率）
- 三态熔断器自动隔离故障模型
- 模型故障时无感知切换到健康模型
- 故障模型恢复后自动重新加入路由

当网关不可用时，中间件回退到静态降级模式。

### 6.3 模型调用重试（ModelRetryMiddleware）

对模型调用失败进行快速重试。采用保守策略（1 次重试、短延迟），因为底层 HTTP 层已有重试机制，这里只做兜底。

### 6.4 工具调用限制（ToolCallLimitMiddleware）

全局限制工具调用次数，防止模型陷入调用循环。同时对特定昂贵工具（如联网搜索）设置更严格的独立限制。

### 6.5 工具调用重试（ToolRetryMiddleware）

对工具调用失败进行指数退避重试（最多 3 次），提升系统鲁棒性。

### 6.6 框架内置中间件

DeepAgents 框架还自动注入以下中间件（项目不做额外配置）：
- **TodoList**：模型可以维护任务清单，跟踪复杂任务的执行进度
- **Filesystem**：模型可以读写文件（通过后端抽象）
- **SubAgent**：将 Specialist 列表注入为可调用的子智能体
- **Summarization**：上下文超长时自动压缩历史消息
- **Memory**：持久化记忆，跨会话保留关键信息
- **HumanInTheLoop**：在配置了 interrupt_on 的工具被调用时挂起执行，等待人类决策

## 七、后台任务执行引擎（Harness）

### 7.1 设计动机

有些复杂任务需要几分钟甚至更长时间才能完成（如招聘流程涉及多轮搜索筛选、薪资审批等）。不能让 HTTP 请求一直等待。因此，复杂任务被转为后台异步执行——用户立即拿到任务编号，后续通过查询接口获取进度。

### 7.2 核心组件

#### TaskExecutor（任务执行器）

后台任务的生命周期管理者。核心职责：
- **任务提交**：接收任务目标，创建独立的执行线程（每个任务是一个独立的 LangGraph 线程），立即返回任务句柄
- **驱动执行**：在后台异步任务中驱动 Executor DeepAgent 逐步执行
- **处理挂起**：当 Agent 触发审批中断时，将状态持久化并休眠，等待外部恢复信号
- **任务恢复**：收到人审决策后唤醒执行线程，继续执行
- **生命周期广播**：任务的创建、执行、挂起、完成、失败等事件通过事件总线实时推送

#### 任务状态机

每个后台任务经历以下状态流转：

```
created → executing → waiting_human → executing → completed
                   ↘ failed / cancelled
```

- **created**：任务已提交，等待执行引擎接管
- **executing**：Executor DeepAgent 正在执行中
- **waiting_human**：执行因审批而挂起，等待人类决策
- **completed**：所有步骤完成，结果已写入对话存储
- **failed**：执行异常终止，错误信息记录在任务句柄中
- **cancelled**：被用户或系统取消

#### 审批兜底机制（P0 安全机制）

正常情况下，Executor 在 Specialist 返回人审标记后的下一轮就会调用审批工具。但 LLM 可能因为幻觉、上下文压缩或推理偏差而连续忽略该标记。为此系统设置了兜底检测：

- 每轮执行后检测是否有新的审批标记出现但未被处理
- 如果连续 3 轮都出现这种情况，强制将任务转入等待人工状态
- 同时保存快照，确保重启后可恢复

这一机制确保了即使模型行为异常，人审流程也不会被绕过。

#### 任务快照与恢复

服务器重启或崩溃后，未完成的任务需要自动恢复执行。系统通过以下机制保障：
- 任务挂起时自动保存快照（包含任务目标、当前状态、审批上下文等）
- 启动时扫描所有快照，根据状态决定恢复策略：等待人工的只注册句柄、执行中的重建后台任务
- 已完成或已终止的任务的快照自动清理

#### 执行日志（Journal）

每次任务执行过程中，系统自动记录关键事件到结构化日志：
- Specialist 委托结果（委托了谁、结果摘要）
- 关键决策点（AI 做出了什么判断）
- 错误事件（什么出错了）
- 审批请求（何时发起了审批）

这些日志在任务恢复时提供上下文，让 Executor 知道"之前发生了什么、现在该做什么"。

#### EventBus（事件总线）

系统内的事件传播机制。基于 Redis 的发布/订阅模式实现（Redis 不可用时降级为内存模式）。任务的每一次状态变更都通过事件总线广播，前端通过 SSE 端点订阅实时更新，无需轮询。

#### DeadLetterQueue（死信队列）

关键操作失败时（如审批数据写入存储失败），操作数据被写入死信队列而非直接丢弃。后台扫描器定期尝试重试，确保重要的副作用操作最终一致。

#### TaskContextManager（上下文管理器）

管理任务的多层记忆：当前轮的热数据、会话级的温数据、以及持久化的冷数据。同时负责快照的保存、读取和清理。

### 7.3 优雅关闭

系统关闭时有一套严格的顺序：
1. 停止接受新任务（排干模式）
2. 等待运行中任务完成（有超时限制）
3. 超时未完成的任务自动保存快照后取消
4. 停止各类后台服务（死信扫描、热加载、健康探活）
5. 关闭数据库连接池

这套顺序确保不会丢失正在执行的任务进度。

## 八、智能模型网关

### 8.1 设计动机

企业级应用需要高可用性。单一模型可能因 API 限流、网络故障、服务降级等原因不可用。模型网关将模型选择从业务逻辑中解耦，提供统一的模型访问层。

### 8.2 核心能力

#### 多模型注册

网关维护一个模型注册表。每个模型按名称、提供商、角色（聊天、检索改写等）、接口类型（LangChain / LlamaIndex）注册。同一模型可以注册多个角色和接口。

#### 角色化路由

系统定义了几种标准角色：
- **CHAT**：用于 Agent 对话和推理
- **FALLBACK_CHAT**：主模型不可用时的备选
- **RETRIEVAL_LLM**：用于检索任务
- **RETRIEVAL_REWRITER**：用于查询改写

消费者按角色请求模型，网关返回健康排序的模型链，消费者无需关心底层是哪个模型、是否发生了切换。

#### 健康跟踪

每次模型调用后，网关记录其延迟和成败状态。这些数据用于实时健康评估和路由决策。

#### 三态熔断器

每个模型有一个独立的熔断器，包含三种状态：
- **关闭（Closed）**：正常状态，请求正常通过
- **半开（Half-Open）**：冷却期后的探测状态，允许少量请求试探模型是否恢复
- **打开（Open）**：熔断状态，请求直接拒绝，避免雪崩

熔断策略：连续失败达到阈值后熔断，进入冷却期，冷却期后进入半开状态进行探测。

#### 后台健康探活

网关启动后台定时任务，以固定间隔（默认 30 秒）对所有注册模型发送轻量级探测请求，及时发现模型恢复。探测任务设计为可快速响应取消，不会阻塞系统关闭。

#### 降级链配置

每个角色可以配置降级链——当主模型不可用时依次尝试哪些备选模型。降级链支持在运行时动态调整。

#### 热切换

管理员可以通过 API 手动激活或切换模型，无需重启服务。熔断器也可以手动重置。

## 九、人机协同（Human-in-the-Loop）

### 9.1 设计理念

有些决策不适合完全自动化——比如薪资审批、敏感数据访问授权、高风险操作确认。系统通过"中断-恢复"模式实现人机协同：Agent 执行到需要人类决策的环节时自动挂起，等待人类做出决策后继续执行。

### 9.2 审批流程

1. Specialist 执行任务过程中发现需要审批的操作，在输出中标记 `[HUMAN_APPROVAL_REQUIRED]`，附带审批上下文
2. Executor 检测到标记，调用 `request_approval` 工具
3. HumanInTheLoop 中间件拦截该工具调用，通过 LangGraph 的 interrupt 机制挂起执行
4. 系统保存任务快照，更新任务状态为"等待人工"，通过事件总线推送通知
5. 人类通过 API 端点做出决策（批准或拒绝）
6. 系统恢复执行，将决策传递给 Agent，Agent 继续执行或调整方案

### 9.3 审批兜底

如第七章所述，系统实现了多层兜底检测，确保即使在 LLM 行为异常时，审批流程也不会被绕过。

## 十、热加载系统

### 10.1 设计动机

在传统的 Agent 系统中，新增一个工具或专家需要修改代码、重启服务。热加载系统让这些变更可以运行时生效，极大降低了运维成本和迭代周期。

### 10.2 工具热加载

监听 `app/tools/` 目录的文件变更事件。当检测到新增、修改时：
1. 等待文件写入稳定（防抖）
2. 触发 Agent 重建流程
3. 原子性地替换运行中的 Agent 实例

### 10.3 SubAgent 热加载

监听 `app/subagents/` 目录的变更事件。当检测到 `AGENT.md` 的新增、修改或删除时：
1. 重新扫描所有 Specialist 定义
2. 重新生成 Triage 和 Executor 的系统提示词
3. 重建两个 Agent 实例
4. 原子性替换，同时更新 TaskExecutor 中的引用

### 10.4 防抖与并发控制

工具和 SubAgent 的热加载可能几乎同时触发（比如一次部署同时更新了多个文件）。系统使用 3 秒防抖窗口和异步锁来合并重建请求，避免短时间内多次完整重建带来的性能冲击。

### 10.5 失败回滚

重建过程中的任何失败都不会影响正在运行的 Agent。如果 Triage 重建成功但 Executor 重建失败，两者都会回滚到旧版本。这确保了热加载的原子性。

## 十一、记忆系统

### 11.1 三层记忆模型

系统采用三层记忆架构，平衡上下文窗口限制与信息持久性：

- **热记忆（Hot）**：当前对话轮次的消息历史，在 LangGraph 状态中直接维护
- **温记忆（Warm）**：会话级别的摘要和关键信息，通过 Summarization 中间件自动压缩历史消息
- **冷记忆（Cold）**：跨会话的持久化记忆，存储在 `/memories/AGENTS.md` 命名空间下，系统可以在多轮对话之间保留用户偏好、历史决策等关键信息

### 11.2 任务执行日志

后台任务的执行日志（Journal）也是一种记忆形式——它记录了任务执行过程中的每一步关键事件。当任务因服务器重启而恢复时，日志摘要被注入到执行上下文中，让 Agent 能够无缝续接。

## 十二、上下文追踪

系统使用 `contextvars` 实现全链路的追踪 ID 传播。每个请求进入时生成一个追踪 ID，该 ID 自动注入到日志记录中。当 Triage 创建后台任务时，追踪 ID 作为父级传递给任务——形成父子链路，使得跨请求、跨线程的调用关系可追溯。

## 十三、安全设计

### 13.1 多租户隔离

每个用户在注册时自动获得独立的资源命名空间。数据在物理层面隔离，不同用户之间无法访问彼此的数据。

### 13.2 认证与授权

所有对话和任务端点需要 JWT 认证。管理端点额外需要管理员令牌。

### 13.3 速率限制

基于滑动窗口算法的请求限流。Redis 可用时使用分布式限流，Redis 不可用时自动降级为内存模式（单进程限流）。

### 13.4 密钥加密

敏感的 API 密钥和数据库凭据在存储前加密，运行时解密使用。

---

## 十四、可观测性

### 14.1 日志

结构化日志，每条日志自动注入追踪 ID 和任务上下文。不同级别（DEBUG / INFO / WARNING / ERROR / CRITICAL）覆盖从正常操作到系统异常的完整频谱。

### 14.2 事件流

任务的状态变更通过事件总线（SSE）实时推送给前端。前端无需轮询即可获得任务进度的即时更新。

### 14.3 追踪集成

集成 Langfuse 和 OpenTelemetry，提供 Agent 执行的完整调用链追踪、延迟分布、成功率统计等可观测性数据。

### 14.4 健康端点

提供标准健康检查端点，用于负载均衡器和监控系统的探活。

## 十五、扩展指南

### 15.1 新增领域专家

1. 在 `app/subagents/` 下创建新目录
2. 编写 `AGENT.md`，声明名称、描述、工具列表和系统提示词
3. 系统自动发现并注入——无需修改任何代码，无需重启

如需新增专家专属工具，按 15.2 节流程添加工具后，在 `AGENT.md` 的 `allowed_tools` 中引用即可。

### 15.2 新增工具

1. 在 `app/tools/` 下创建工具文件
2. 使用注册装饰器标注工具函数
3. 工具自动进入注册中心——热加载器检测到后自动重建 Agent

### 15.3 新增模型

1. 在配置文件中声明模型的提供商、API 密钥、角色
2. 系统启动时自动加载并注册到模型网关
3. 可通过管理 API 运行时调整模型状态

## 十六、关键设计决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 路由方式 | LLM 自路由 | 避免硬编码路由规则，模型可根据上下文灵活决策 |
| Agent 分层 | 两层（Triage + Executor） | 将快速响应与长周期执行分离，各层独立优化 |
| 专家定义 | 文件系统驱动（AGENT.md） | 零代码扩展，热加载自动发现 |
| 中间件架构 | 管线模式 | 横切关注点分离，可组合，可替换 |
| 任务执行 | 异步后台 + 中断恢复 | 长周期任务不阻塞 HTTP，支持人机协同 |
| 模型管理 | 智能网关 + 熔断器 | 高可用，健康感知路由，零停机切换 |
| 记忆模型 | 三层（热/温/冷） | 平衡上下文窗口限制与信息持久性 |
| 事件传播 | Redis Pub/Sub + 内存降级 | 实时推送，有损降级不影响核心功能 |
| 容错 | 快照恢复 + 死信重试 + 审批兜底 | 多重保障，确保任务不丢失、审批不绕过 |
| 热加载 | 文件监听 + 防抖 + 原子替换 + 失败回滚 | 零停机演进，安全可靠 |
