# TK-MultiAgent — 项目经历总结

---

## 项目描述

**TK-MultiAgent** 是一个面向企业招聘场景的 Multi-Agent 智能协作系统。系统基于 FastAPI + LangGraph + DeepAgents 构建，实现了三层 DeepAgent 编排架构（Triage 分流 → Executor 后台执行 → Specialist 领域专家），支持复杂长周期任务的自主规划、多 Agent 协作、Human-in-the-Loop 审批流程，以及 LLM 零停机热切换的智能模型网关。

**技术栈**：Python / FastAPI / LangGraph / DeepAgents / LlamaIndex / LangChain / PostgreSQL / Milvus / Redis / MinIO / Langfuse

**项目规模**：~80 个模块，16 个 Agent 工具，8 个领域 Specialist，4 层中间件管线

---

## 核心工作与成果

### 1. 三层 Multi-Agent 编排架构（自主设计）

**背景**：现有 Agent 框架大多只支持单层 Agent 或简单的静态路由。对于企业招聘场景——需要多个领域专家协作、涉及人审流程、任务执行周期从秒级到小时级不等——需要一套能自动分流的编排机制。

**方案**：
- 设计 **Triage → Executor → Specialist** 三层架构，LLM 通过 Function Calling 自路由（无需外部分类器）
- Triage 层负责意图理解和任务分流（简单直接处理 / 复杂转后台）
- Executor 层作为 LLM 驱动的后台执行引擎，自主规划、逐步委托、动态调整
- 8 个 Specialist 通过声明式 `AGENT.md` 定义，启动时自动发现并注入路由表

**成果**：
- 新增 Specialist 只需编写一个 Markdown 文件，零代码侵入
- LLM 自路由方案相比外部分类器减少了级联错误和维护成本
- 支持从"现在几点"到"帮产品部招一个高级后端工程师并完成全流程审批"的弹性任务范围

### 2. Human-in-the-Loop 审批流（解决多层 interrupt 传播难题）

**背景**：LangGraph 的嵌套 interrupt 在多层 Agent 架构中不可靠传播——在 Specialist（子图）内部触发 `interrupt()` 时，外层 Executor 可能无法正确捕获。

**方案**：
- 设计**双层分离**审批模式：Specialist 只负责写入 Store + 返回标记字符串，Executor 检测标记后触发 `interrupt()` 挂起
- 实现 **P0 级别审批兜底检测**：连续 3 轮检测到审批标记但 LLM 未处理时，强制合成 interrupt 将任务转入等待状态，防止 LLM 幻觉导致审批被跳过
- 支持**薪资分级审批规则**：≤30K 用人经理审批 → 30-50K 部门负责人加签 → >50K CEO 终审

**成果**：
- 消除了多层 Agent 嵌套 interrupt 的竞态和死锁问题
- P0 兜底机制确保即使 LLM 出现推理偏差，审批流程也不会被静默跳过
- 审批数据持久化到 Store，服务重启后审批状态不丢失

### 3. 智能模型网关（零停机热切换 + 熔断降级）

**背景**：生产环境中 LLM API 偶尔不可用，需要自动降级和恢复能力，管理员需要在不重启服务的情况下切换模型。

**方案**：
- 实现 **ModelGateway 中心单例**：统一管理所有模型实例的注册、路由、健康跟踪
- 实现**三态熔断器**（CLOSED → OPEN → HALF_OPEN）：5 次连续失败熔断 / 30s 冷却 / 探测恢复
- 实现**健康感知路由**：GatewayMiddleware 根据模型健康状态自动选择降级链
- 实现**后台定期探活**（30s 间隔）：自动检测模型恢复并关闭熔断
- 支持**零停机热切换**：管理员通过 API 切换模型，进行中的请求不受影响

**成果**：
- 主模型故障时自动降级到备用模型，用户无感知
- 模型恢复后自动切回，响应延迟恢复至正常水平
- 使用 per-model `asyncio.Lock` 解决 GatewayMiddleware 和 HealthProbe 并发写入 HealthRecord 的竞态条件

### 4. 后台任务执行引擎（长周期任务生命周期管理）

**背景**：复杂招聘任务（如"筛选本月所有 P7 候选人并做匹配度分析"）需要分钟到小时级的执行时间，不能阻塞 HTTP 请求-响应。

**方案**：
- 实现 `TaskExecutor`：以 `asyncio.Task` 在后台驱动 LangGraph `astream()`，与 HTTP 请求-响应解耦
- 实现**任务快照 + 重启恢复**：未完成任务在服务关闭时自动保存快照到 Store，重启后自动重建执行线程
- 实现**执行日志（Journal）**：结构化事件记录（specialist_result / decision / approval / error / completed），不受上下文压缩影响
- 实现**任务结果回写**：后台任务完成后自动将结果写入对话 Store，Triage 在下一次用户对话时引用
- 实现**优雅关闭**：排干新任务 → 等待运行中任务完成 → 超时后保存快照 → cancel

**成果**：
- 支持服务重启后自动恢复所有未完成任务，包括正在等待人审的任务
- Journal 机制解决了 LLM 上下文窗口压缩后执行信息丢失的问题
- 三层记忆模型（Hot/Warm/Cold）管理执行过程中的上下文生命周期

### 5. 全链路追踪与可观测性

**方案**：
- 基于 Python `contextvars` 实现 trace_id 全链路传播，无需显式传参
- 链路格式 `trace-{uuid16}/task-{task_id_short}`，前后台自动继承
- `TraceIdFilter` 自动为每条日志注入 trace_id
- SSE 端点实时推送任务状态事件（created / executing / interrupted / completed / failed）
- `StatusCallbackHandler` 推送 Agent 运行时状态（工具调用 / 模型降级 / 上下文压缩）
- 集成 Langfuse 进行 LLM 调用追踪

### 6. 中间件管线与韧性设计

**方案**：
- 组装 8 层中间件管线：RateLimit → Gateway（健康路由）→ ModelRetry → ToolLimit ×2 → ToolRetry
- 实现**死信队列**（DeadLetterQueue）：关键操作（审批 Store 写入）失败后持久化重试，指数退避（1m→2m→4m→max 1h）
- 实现**事件总线**（EventBus）：Redis PubSub + 本地 handler 双通道，Redis 不可用时自动降级为内存模式
- 幂等设计贯穿全局：任务创建、审批请求、文档索引、死信入队均使用 SHA-256 确定性键

### 7. 工具系统与 Schema 注册中心

**方案**：
- 设计 `TOOL_REGISTRY` + `@register_tool` 装饰器的工具注册机制，工具名冲突自动检测
- 16 个 Agent 工具按 Specialist 的 `allowed_tools` 声明精确分发，无关工具不注入（减少 token 消耗）
- 实现 `SchemaRegistry`：JSON Schema → Pydantic 动态模型，支持内联声明和命名引用两种方式
- 结构化输出使下游 Agent 可精确引用上游数据，无需文本正则提取

## 技术亮点

| 维度 | 实现 |
|------|------|
| **Agent 编排** | 三层 DeepAgent + LLM Function Calling 自路由，无外部分类器 |
| **人审流程** | 双层分离审批模式 + P0 连续3轮兜底检测，消除嵌套 interrupt 死锁 |
| **模型网关** | 多模型注册/熔断/降级/热切换 + 后台探活自动恢复 |
| **韧性设计** | 四层防御（审批兜底→死信重试→熔断降级→重试中间件）+ 幂等去重 |
| **任务恢复** | 快照持久化 + 服务重启自动恢复，Journal 结构化日志抗上下文压缩 |
| **可观测性** | contextvars 全链路 trace + SSE 实时推送 + Langfuse 集成 |
| **声明式扩展** | AGENT.md 零代码新增 Specialist，@register_tool 自动注册工具 |
| **并发安全** | asyncio.Lock 保护共享状态（HealthRecord / DeadLetterQueue / CircuitBreaker） |
