# 实时状态系统设计方案

## 1. 现状

### Agent 聊天管线（`/chat` SSE）

当前 `event: message` 只推 token 内容，`event: sources` 最后推引用来源。Agent 内部执行过程是黑箱：

```
用户发送消息
    ↓
agent.astream()  ← 整个过程不可见
    ↓
SSE: event=message  data={"content": "..."}     ← 用户直接看到文字
SSE: event=sources  data={"sources": [...]}      ← 最后看到来源
```

**问题：** LLM 推理 1-10s、知识检索 1-5s、SubAgent 执行 5-30s 这些等待期间，用户看到的是空白。

### 文档处理管线（`/documents/events` SSE）

当前状态粒度只有两级：

```
update_document_status(doc_id, status="processing")   ← 开始
    ↓  ... 5-30 秒黑箱 ...
update_document_status(doc_id, status="indexed")       ← 完成
update_document_status(doc_id, status="failed")        ← 失败
```

**问题：** 清洗卡了还是切分卡了还是写 Milvus 卡了，无从得知。

## 2. 总体架构

沿用现有的两套 SSE 通道，不做新基建：

```
┌─ Agent 聊天 ──────────────────────────────────────────────┐
│                                                           │
│  POST /chat                                               │
│    ↓                                                      │
│  generate_response_stream()                               │
│    ├─ agent.astream(..., stream_mode="messages")          │
│    │       ↓ 原有产出                                        │
│    │    event: message  data: {"content": "..."}          │
│    │    event: sources  data: {"sources": [...]}          │
│    │                                                      │
│    └─ StatusCallbackHandler  →  asyncio.Queue             │
│            ↓ 新增产出                                     │
│         event: status   data: {"status": "thinking",      │
│                                "label": "思考中...",      │
│                                "timestamp": "..."}         │
│                                                           │
│  两个产出在 event_generator() 里用 asyncio.wait 多路复用     │
│                                                           │
│  协议: /chat 一次请求的生命周期内，收到 n 条 status +       │
│        若干条 message + 可选的 sources                     │
└───────────────────────────────────────────────────────────┘

┌─ 文档处理 ─────────────────────────────────────────────────┐
│                                                           │
│  POST /upload → BackgroundTasks                           │
│    ↓                                                      │
│  process_and_index()   ← 在关键步骤插入 publish           │
│    │                                                      │
│    └─ pg.update_document_status()                         │
│            ↓ 内部调 document_event_bus.publish()          │
│                                                           │
│  EventSource('/documents/events')                          │
│    event: status_change  data: {"doc_id": N,              │
│                                   "status": "parsing",    │
│                                   "filename": "xxx.pdf"}  │
│                                                           │
│  协议: 一次上传产生一组 status_change 事件，以 indexed      │
│        或 failed 结尾                                     │
└───────────────────────────────────────────────────────────┘
```

## 3. Agent 运行时状态

### 3.1 所有状态枚举

按 Agent 执行流程从前到后：

```
thinking            思考中...          LLM 首轮推理（选工具）
retrieving          知识检索中...      knowledge_query 工具执行
searching           联网搜索中...      web_search 工具执行
executing_task      执行任务：{技能名}   task 工具（SubAgent 委托）
waiting_subagent    等待 SubAgent...   SubAgent 执行中
generating          生成回答中...      LLM 第二轮推理（生成最终回答）

retrying            重试中...         模型/工具重试（第 N 次）
degraded            已降级备用模型      Gateway 降级到 Qwen
timeout             执行超时           超出时间限制
cancelled           用户已取消          前端断开连接

completed           完成               SSE 流正常结束
```

### 3.2 实现：StatusCallbackHandler

新建 `app/status_handler.py`：

```python
class StatusCallbackHandler(AsyncCallbackHandler):
    """捕获 LangGraph 执行过程中的关键事件，推入队列。"""

    def __init__(self, queue: asyncio.Queue):
        self.queue = queue
        self._thinking_sent = False

    async def on_llm_start(self, serialized, prompts, **kwargs):
        if not self._thinking_sent:
            await self.queue.put(("thinking", "思考中..."))
            self._thinking_sent = True

    async def on_tool_start(self, serialized, input_str, **kwargs):
        name = serialized.get("name", "unknown")
        if name == "async_knowledge_query_ask":
            await self.queue.put(("retrieving", "知识检索中..."))
        elif name == "async_web_search":
            await self.queue.put(("searching", "联网搜索中..."))
        elif name == "task":
            # 解析 SubAgent 名称（从 input_str 中提取）
            agent_name = self._extract_subagent_name(input_str)
            await self.queue.put(
                ("executing_task", f"执行任务：{agent_name}")
            )

    async def on_tool_end(self, output, **kwargs):
        # 工具完成后若之前发过 retrieving，切回 thinking
        await self.queue.put(("thinking", "思考中..."))

    async def on_retry(self, retry_state, **kwargs):
        attempt = retry_state.attempt_number
        await self.queue.put(("retrying", f"重试中（第 {attempt} 次）"))

    async def on_llm_end(self, response, **kwargs):
        if self._thinking_sent:
            await self.queue.put(("generating", "生成回答中..."))
```

### 3.3 多路复用：修改 generate_response_stream

`main.py` 的 `generate_response_stream` 改为同时监听两个源：

```python
async def generate_response_stream(...) -> AsyncGenerator[str, None]:
    queue = asyncio.Queue()
    handler = StatusCallbackHandler(queue)
    configurable["callbacks"].append(handler)  # 追加到现有 callbacks

    async def agent_stream():
        """消费 agent.astream，产出 content 和 error"""
        try:
            async for chunk in agent.astream(..., stream_mode="messages"):
                if isinstance(chunk, (tuple, list)) and len(chunk) > 0:
                    msg_chunk = chunk[0]
                    if isinstance(msg_chunk, AIMessageChunk) and msg_chunk.content:
                        yield json.dumps({"content": msg_chunk.content, "session_id": session_id})
        except Exception as e:
            yield json.dumps({"error": str(e), "session_id": session_id})
        finally:
            await queue.put((None, None))  # 哨兵

    async def status_stream():
        """消费队列，产出 status"""
        while True:
            status, label = await queue.get()
            if status is None:
                break
            yield json.dumps({"status": status, "label": label, "session_id": session_id})

    # 多路复用
    async for event in _merge_streams(agent_stream(), status_stream()):
        yield event
```

`_merge_streams` 用 `asyncio.wait(..., FIRST_COMPLETED)` 实现：

```python
async def _merge_streams(agent_gen, status_gen):
    """两个异步生成器多路复用，谁先到就 yield 谁。"""
    async def _collect(gen, results):
        async for item in gen:
            results.append(item)
            return  # 每次只取一条，让出控制权
        results.append(None)  # 结束

    agent_done, status_done = False, False
    agent_results, status_results = [], []

    while not (agent_done and status_done):
        tasks = []
        if not agent_done:
            tasks.append(asyncio.ensure_future(_collect(agent_gen, agent_results)))
        if not status_done:
            tasks.append(asyncio.ensure_future(_collect(status_gen, status_results)))

        if not tasks:
            break

        done, _ = await asyncio.wait(tasks)

        for t in done:
            if t in tasks[:1] if tasks else []:  # agent task
                if agent_results:
                    item = agent_results.pop(0)
                    if item is None:
                        agent_done = True
                    else:
                        yield item
            else:  # status task
                if status_results:
                    item = status_results.pop(0)
                    if item is None:
                        status_done = True
                    else:
                        yield item
```

## 4. 文档处理状态

### 4.1 所有状态枚举

按 `process_and_index` + `process_document` 的执行顺序：

```
uploading         上传中...                 upload_routes 收到请求
deduplicating     哈希校验中...             SHA-256 计算
storing           存储到 MinIO...          put_object
queuing           排队中...                BackgroundTasks 等待调度
downloading       下载文档中...             process_and_index → fget_object
loading           解析文档中...             SimpleDirectoryReader.load_data
                   ├─ pdf → PyMuPDFReader
                   ├─ xlsx → PandasExcelReader
                   ├─ docx → DocxReader
                   └─ txt/md/html → 默认
cleaning          清洗数据中...             DataCleaningComponent
splitting         切分节点中...             NodeParser
                   ├─ 语义切分（大文件）
                   ├─ 按行切分（表格）
                   └─ 按段落（文本）
indexing          写入向量库...             ainsert_nodes
                   └─ 向量化 + Milvus 写入
indexed           索引完成 ✓               成功
failed            索引失败 ✗               异常
```

### 4.2 埋点位置

**`index_manager.py`** — `process_and_index` 方法：

```python
async def process_and_index(self, ...):
    tmp_path = None
    try:
        await self.pg.update_document_status(doc_id, status="queuing")

        # 从 MinIO 下载
        await self.pg.update_document_status(doc_id, status="downloading")
        await minio_client.fget_object(...)

        # 获取文件哈希
        doc = await self.pg.get_document(doc_id)
        file_hash = doc["file_hash"] if doc else None

        # 处理管线
        await self.pg.update_document_status(doc_id, status="loading")
        nodes = await process_document(
            file_path=tmp_path, ..., file_hash=file_hash,
        )

        # 写入 Milvus
        await self.pg.update_document_status(doc_id, status="indexing")
        index = await async_get_milvus_index(user_id=user_id, embed_model=embed_model)
        await index.ainsert_nodes(nodes)

        await self.pg.update_document_status(
            doc_id, status="indexed", chunk_count=len(nodes),
        )
    except Exception as e:
        await self.pg.update_document_status(doc_id, status="failed", error_message=str(e))
```

**`document_processor.py`** — `process_document` 函数（注：这里没有 pg 引用，需要改签名或传回调）：

前端将两路 SSE 的状态渲染到同一个组件，位置在上传按钮附近：

```
│  [+ 上传文档]                             │
│                                           │

│                                           │
│  📄 2024年报.pdf                          │
│  ├── 下载文档中...                        │
│  ├── 解析文档中...                        │
│  ├── 清洗数据中...                        │
│  ├── 切分节点中...   ── 最新一条高亮 ──→   │
│  └── 索引完成 ✓                          │
│                                           │
│  💬 对话 #127                             │
│  ├── 思考中...                            │
│  ├── 知识检索中...                        │
│  ├── 思考中...                            │
│  └── 生成回答中...                        │
│                                           │
│  📄 Q3财报.xlsx                           │

│                                           │
│  最新: 正在切分节点中...   (闪烁)         │

```

两种布局方式：

1. **列表模式**（如上图）：按文档/会话分组，每个条目显示该次操作的状态链
2. **单行模式**：只显示最新一条状态，点击展开详情

建议实现方式：前端维护一个 `Map<doc_id, StatusEvent[]>` 和一个 `Map<session_id, StatusEvent[]>`，收到 SSE 事件后 append 到对应数组，状态框实时刷新。

## 8. 关键设计决策

1. **Callback 方案 vs 直接改图节点**：Callback 方案侵入最小（不改 Agent 编译图），缺点是状态粒度受限于 LangChain 的 callback 事件类型。如果将来需要更细的 SubAgent 内部状态，可以加自定义 callback。

2. **文档状态在 process_and_index 里埋点 vs 传到 process_document 里埋点**：选择在 `process_and_index` 里埋，因为它是编排层，`process_document` 是纯计算层不应耦合持久化。`loading`/`cleaning`/`splitting` 三个状态在调用 `process_document` 前后设即可。

3. **前端展示**：只负责接收渲染，不缓存/组合状态逻辑。两个 EventSource 各管各的，前端只做 append + 最新高亮。

4. **两套状态通道完全独立**：Agent 状态（`/chat` SSE）和文档状态（`/documents/events` SSE）互不依赖。Agent 不触发文档操作，文档只由用户手动上传触发。前端状态框同时监听两个 EventSource，按来源分组展示即可。
