儒家思想：集百家之长
企业级RAG本质上从传统数据库存储提取转化为向量或语义实体相关联的外挂式向量数据库，采用对话交互模式，来获取用户想要的信息
怎么才是实现一个合格的生产级别的RAG
使用的场景：
    1.医疗
    2.法律
    3.集团公司财务
    4.工业制造业手册
    5.客服
    6.教育
我们需要关注的：
    原始文件上传到minio中去 使用miniopy-async
    针对pdf相关文件: 可以分层设计解析，复杂的、扫描的用LlamaParse来分析，普通的PyMuPDFReader来分析。也可以在前端让用户自己判断选择。
    索引的管理：
    插入
    删除
    更新
    刷新
    1.原始数据从来源到结构化 构建数据管道Ingestion Pipeline 再按照 TransformComponent 列表顺序执行，完成数据分析、清洗、元数据提取。
    2.动态切分怎么根据原始数据来实现合适的切分策略
    3.多路构建索引常规VectorStoreIndex,多模态MultiModalVectorStoreIndex
    4.动态索引管理
    4.检索的时候如何动态检索，智能路由，尽可能的实现快速检索、准确检索
    5.生成时，如何控制上下文
    6.在检索的时候怎么评估
    7.在生成的时候怎么评估
    8.混合文档（文字，图片，表格）
    9.检索延迟问题，首字延迟
    10.长下文模型的注意力稀释幻觉
    11.知识原文溯源
解决方案：
    引入知识图谱、多模态qwen3-vl-embedding qwen3-vl-rerank
    分层协同架构：
    1.LlamaParse解析文本、图片、表格，保留它们的上下文关系（元数据）
    2.多模态知识和向量索引协同处理
        1.知识图谱构建：从文档中抽取核心实体和关系，构建知识图谱来关联它们
        2.将图谱的实体和关系以及文本块，视觉元素分别生成向量，存入向量库，实现跨模态的语义对齐与检索
    3.智能路由、动态检索，意图识别：事实推理、关系推理、视觉查询 可元数据过滤
在源数据分析阶段从document到node中间自定义IngestionPipeline加上元数据提取是为匹配意图识别提高召回率
元数据提取特征提取器：
    1.SummaryExtractor - 自动提取一组节点的摘要
    2.QuestionsAnsweredExtractor - 提取每个节点可以回答的一组问题
    3.TitleExtractor - 提取每个节点上下文的标题
    4.EntityExtractor - 提取每个节点内容中提到的实体（即地点、人物、事物的名称）
动态索引管理：
    1.用户直接上传模式：
        用户上传后，计算上传文件哈西，保留哈西为元数据，构建数据管道，进行分析清洗，元数据提取，选择切分策略，最终构建成node
        在这个阶段我们需要进行去重，删除，跳过，重建，新建
    2.前端页面展示知识库管理界面：
        新建一个路由，专门用于知识库管理界面
        用户在前端点击知识库管理后默认展示他上传了的前20个文件的文件列表，包含文件名，文件类型，上传时间，文件大小
        在这个界面可以模糊搜索文件名，然后可以进行替换，删除，查看等操作
    3.由于我整个系统都是用户隔离的，所以每个用户都只能操作属于自己的知识库
